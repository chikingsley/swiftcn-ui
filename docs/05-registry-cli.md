# Distribution: registry + CLI (`swiftcn add button`)

shadcn's real moat is not the components — it's the distribution machine: a flat-file registry schema plus a CLI that copies code *into* your project. As of June 2026, **any public GitHub repo containing a `registry.json` is a valid shadcn registry** — no server, no publishing step. That model ports to Swift almost 1:1 and is the cheapest possible v1 for us.

This is Phase 5 work — nothing here blocks component development, but the file anatomy rules in `02-architecture.md` (self-contained files, Theme-only dependency) exist so this layer can be added without rewrites.

## The schema, ported

### `registry.json` (repo root)

```json
{
  "$schema": "https://swiftcn.dev/schema/registry.json",
  "name": "swiftcn",
  "homepage": "https://swiftcn.mintlify.app",
  "items": [ … ]
}
```

### Registry item (per component)

```json
{
  "name": "button",
  "type": "registry:component",
  "title": "Button",
  "description": "Displays a button. 6 variants, 4 sizes.",
  "files": [
    { "path": "Sources/Swiftcn/Components/Button.swift", "target": "Swiftcn/Button.swift" }
  ],
  "registryDependencies": ["theme"],
  "platforms": { "iOS": "17.0", "macOS": "14.0" },
  "categories": ["form"]
}
```

Differences from shadcn's schema, driven by Swift realities:

- `dependencies` (npm) → **SPM package references** (rarely needed; e.g. Charts wrappers use only Apple frameworks). Something like `"spmDependencies": [{ "url": "…", "from": "1.0.0" }]` — the CLI can edit `Package.swift` or just print instructions for Xcode projects.
- `platforms` replaces web-framework concerns.
- `cssVars` → a `theme` item type that copies the `Theme/` folder; component items depend on `"theme"` the way shadcn items depend on CSS vars being present.
- Blocks are `"type": "registry:block"` with multiple files, exactly like shadcn.

## The CLI

Swift executable (`swift-argument-parser`), distributed via Homebrew tap and Mint:

```
swiftcn init            # copies Theme/ into your project, writes swiftcn.json
swiftcn list            # components in the configured registries
swiftcn view button     # print the source before adding
swiftcn add button card # resolve registryDependencies, copy files
swiftcn add owner/repo/thing   # GitHub-as-registry, third-party components
```

`swiftcn.json` in the consumer project (≙ `components.json`):

```json
{
  "target": "MyApp/UI/Swiftcn",
  "prefix": "SC",
  "registries": { "@acme": "https://github.com/acme/acme-swiftui-registry" }
}
```

Resolution flow (identical to shadcn's): name → registry lookup → fetch item JSON → topo-sort `registryDependencies` → copy files to `target` → report. "Last write wins" on path collisions is the sanctioned override mechanism, same as shadcn.

Two consumer-project realities to handle:

1. **Xcode projects don't auto-include loose files** the way SPM targets do. Files copied into a folder that's a synchronized group (Xcode 16 default) just work; classic `.xcodeproj` file references would need `xcodeproj`-gem-style editing — v1 answer: require/recommend synchronized folders, print a note otherwise.
2. **Renaming the prefix**: `prefix` config lets the CLI rewrite `SC` → user's prefix on copy (plain textual rename is safe given our naming conventions).

## Why bother, given SPM exists?

Same reason shadcn beat installable component libraries: an SPM dependency is a black box you can't edit without forking, and UI components are exactly the code teams always need to tweak. Copy-paste ownership + a tool that automates the copying = the whole thesis. SPM remains offered for people who genuinely don't want to own the code.

## Milestones

1. **M0 — DONE:** `registry.json` checked into this repo (generated by `scripts/generate_registry.py`); humans (and AI agents) use it as a machine-readable index.
2. **M1 — DONE:** `swiftcn` CLI at [`cli/`](../cli/) with `list`/`view`/`add`/`init`: resolves `registryDependencies` transitively (topo-sorted), reads a local registry (walks up from cwd) or raw.githubusercontent, `--dry-run`/`--overwrite`/`--target`, unknown-name suggestions. Run it with `swift run --package-path cli swiftcn …`.
3. **M2:** `swiftcn.json` consumer config, prefix rewriting, third-party GitHub registries, Homebrew/Mint distribution.
4. **M3:** namespaced registries + auth headers (private/team registries — this is where shadcn found its enterprise traction).
